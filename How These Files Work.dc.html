How these files work

Your dashboard is not a classic HTML + CSS + JS site. It is a single Design Component file (.dc.html) plus a shared runtime (support.js). This page explains every moving part and tells you exactly where to edit what. New to JavaScript? Read Part 2 · JavaScript from zero after this overview.

1 · The files in this project

Personal Dashboard.dc.html
The entire app: markup, data, and behavior in one file (~3,000 lines). This is the only file you edit.
Strategy Dashboard.dc.html
An earlier version, kept as backup. Same structure.
support.js
The runtime engine. It reads the template, fills in the {{ ob }} … {{ cb }} holes, runs the logic class, and renders everything with React under the hood. Never edit this file — it is generated and shared by all components.

The .dc.html file opens directly in a browser like any HTML file — support.js just has to sit next to it.

2 · Anatomy of the .dc.html file

Four zones, top to bottom:

<!DOCTYPE html>
<head> … <script src="./support.js"> … </head>      ← ① loader (boilerplate)
<body>
<x-dc>
  <helmet> fonts, theme CSS variables, resets </helmet>   ← ② head content
  … the TEMPLATE: all markup, with {{ ob }} holes {{ cb }} …           ← ③ the view
</x-dc>
<script type="text/x-dc" data-dc-script data-props="…">
  class Component extends DCLogic { … }                    ← ④ the logic
</script>
</body>
① Loader — fixed boilerplate; leave alone.
② <helmet> — the equivalent of <head>: font links, the light/dark theme variables ([data-theme="light"] / [data-theme="dark"]), and body resets. This is the only place a <style> or <script> tag is allowed.
③ Template — everything visible. In your dashboard: lines ~60–1300. All styling is inline (style="…") using the CSS variables from ②; there are no CSS classes.
④ Logic class — one JavaScript class named Component. In your dashboard: lines ~1306 to the end. All data and behavior live here.

3 · The template language

Holes: {{ ob }} name {{ cb }}

A hole is filled with a value the logic class provides. Holes are lookups only — a name or dotted path like {{ ob }} n.label {{ cb }}. You can never put code in one ({{ ob }} a + b {{ cb }} silently renders nothing). Any computation happens in the logic class, which exposes the result under a name.

Holes work in text and in attributes. width:{{ ob }} sbWidth {{ cb }} mixes into the style string; onClick="{{ ob }} togTheme {{ cb }}" passes a function. Event names are camelCase like React (onClick, not onclick).

Conditionals and loops

Instead of writing repeated markup or toggling with JS, the template has two control tags:

<sc-if value="{{ ob }} isHome {{ cb }}" hint-placeholder-val="{{ ob }} true {{ cb }}">
  … rendered only when isHome is true …
</sc-if>

<sc-for list="{{ ob }} navItems {{ cb }}" as="n" hint-placeholder-count="7">
  <div>{{ ob }} n.label {{ cb }}</div>   ← repeated once per item; $index available
</sc-for>

The hint-* attributes are placeholders shown while the page is still loading its values — keep them when copying a block.

Styling

Everything is styled inline via style="…", referencing the theme variables (var(--surface), var(--accent)…). Hover/focus states use special attributes: style-hover="background:var(--accent-muted);". Because both themes define the same variable names, the theme toggle just swaps data-theme on the root div and every color updates.

4 · The logic class

It behaves like a React class component (state, setState, lifecycle) but has no render method. Instead it has renderVals(), which returns a plain object — and every key of that object becomes a hole name the template can use.

class Component extends DCLogic {
  state = { theme: 'dark', nav: 'home', … };   // live UI state

  componentDidMount() { …load saved checkmarks… }  // runs once on load

  plans() { return { studyAbroad: {…}, portfolio: {…}, … } }  // ★ ALL CONTENT

  renderVals() {                     // ★ the bridge to the template
    return {
      theme: this.state.theme,       // → data-theme="{{ ob }} theme {{ cb }}"
      isHome: this.state.nav === 'home',   // computed, exposed by name
      navItems: […],                 // arrays for sc-for loops
      togTheme: () => this.setState({ theme: … })  // click handlers
    };
  }
}

Key structure in your dashboard: the plans() method (~line 1354) holds every word of content as a big data object — plan titles, routes A/B/C, tiers, action items, timeline bars. renderVals() (~line 2172) transforms that data into the flat values and handlers the template consumes.

5 · How data flows

plans() datastatic content: text, dates, structure
↓
statewhat's selected, expanded, checked, which theme
↓
renderVals()combines both into flat named values + handlers
↓
template holes{{ ob }} … {{ cb }} fill in; conditionals and loops expand
↓  (click → handler → setState → renderVals runs again → view updates)

There is no manual DOM manipulation anywhere — no querySelector, no addEventListener. Change state, and the runtime re-renders whatever depends on it.

6 · What gets saved between visits

Three things persist in the browser's localStorage, written by the handlers and re-loaded in componentDidMount():

strategy-dashboard-checks — every checkbox you tick
pf-gate — portfolio gate self-assessment levels
pf-track — selected portfolio track

Everything else (theme, open panels, selected route) resets on reload. Clearing browser site data wipes your checkmarks.

7 · Where to edit what

Change any text, date, or plan content → the plans() data object (from ~line 1354). Titles, routes, tiers, tasks, timeline bars, wiki insights — it is all plain strings and arrays there. Search for the exact text you see on screen.
Change colors / themes → the two [data-theme] variable blocks in <helmet> (top of file). One variable change recolors everything that uses it, in both themes.
Change layout / spacing / typography of a section → find its markup in the template (sections carry data-screen-label attributes like "Home") and edit the inline style="…".
Add a new plan → append an entry in plans() following the existing 3-zone shape, plus a nav item in the NAV list. No new markup needed — the plan view is generic and renders whatever plan is selected.
Change behavior (what a click does, what counts as "today") → the handlers and computations inside renderVals().
Tweakable settings → the data-props JSON on the logic script tag declares props (currently defaultRoute and dimStrength) that surface as editor controls and reach the code as this.props.x.

8 · Rules & gotchas

✕No expressions in holes. {{ ob }} items.length {{ cb }} works (dotted lookup); {{ ob }} count + 1 {{ cb }} renders nothing. Compute in renderVals().
✕No CSS classes or stylesheets in the body — inline styles only. <style> in helmet is reserved for fonts, keyframes, variables, and resets.
✕No <script> tags in the template body. All JS lives in the logic class; run-on-load code goes in componentDidMount().
✕The class must be named exactly Component and extend DCLogic. No import/export — it is plain classic JavaScript.
✓To update state, always use this.setState(…) — never assign to this.state directly, or the view won't refresh.
✓When copying a repeated block, keep the whole loop/conditional wrapper including its hint-* attributes.
✓Before big edits, duplicate the file (as was done with Strategy Dashboard) so you always have a working version to fall back to.
