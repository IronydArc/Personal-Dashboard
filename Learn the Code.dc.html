1 · Values and variables

JavaScript works with values. The three kinds this file uses constantly:

'dark'        ← a string  (text, always in quotes)
210           ← a number   (no quotes)
true / false  ← a boolean  (yes/no switch)

A variable is a named box holding a value. const creates one that won't be re-assigned; let one that can:

const S = this.state;   ← "S" is now a shorthand name for the state
let total = 0;          ← a counter that will change

Spot it in your file: the top of renderVals() starts with const S = this.state; and const PLANS = this.plans(); — just short nicknames so the rest of the code is less wordy.

2 · Objects — labeled boxes inside boxes

An object groups values under labels (called keys). It is written with curly braces:

const route = {
  key: 'A',                    ← key "key", value 'A' (string)
  title: 'CSC Type A',         ← key "title"
  tiers: [1, 2]                ← a key can hold ANY value, even an array
};

You read a value out with a dot: route.title → 'CSC Type A'. Dots chain into nested objects — this is the single most important idea for your file:

PLANS.home.strategy.focus.title
  │     │      │       │     └─ finally: the string itself
  │     │      │       └─ inside that, the "focus" object
  │     │      └─ inside that, the "strategy" object
  │     └─ the "home" object inside PLANS
  └─ the big data object from plans()

Your entire plan content — every route, tier, task — is one giant nested object returned by plans(). Editing content = finding the right key and changing its value. The template's {{ ob }} n.label {{ cb }} holes are exactly these dot-paths.

The alternative bracket form checks[id] does the same as a dot, but the key comes from a variable: if id is 'p2a', then checks[id] means checks.p2a.

3 · Arrays — ordered lists

An array is a list of values in square brackets, in order:

routes: [
  { key: 'A', title: 'CSC Type A · General Scholar' },   ← position 0
  { key: 'B', title: 'CIS · Confucius Institute' },      ← position 1
  { key: 'C', title: 'CSC Type B · Bachelor of AI' }     ← position 2
]

Counting starts at 0, so routes[0] is Route A. routes.length is 3. Arrays of objects like this are the pattern behind every repeated thing on screen: nav items, checklists, timeline bars. The template's sc-for loop walks an array and stamps out one block of markup per item.

To add an item (say, a fourth route): copy one existing { … }, entry, paste it after, and edit its values. Mind the commas — every item except the last needs a trailing comma.

4 · Functions — stored actions

A function is a recipe you define once and run later. Your file uses the compact arrow form:

const togTheme = () => {
  this.setState({ theme: 'light' });
};
      └ name        └ () = "takes no ingredients"   └ the body: what happens when run

Reading an arrow: (inputs) => result — "given these inputs, do/produce this." Two shortcuts you'll see:

  • One input needs no parentheses: r => r.title ("given a route, produce its title").
  • If the body is a single expression, its value is returned automatically — no return, no braces.

Crucially, a function is itself a value. It can sit in an object, be passed to the template, and run only when clicked. That is what onClick="{{ ob }} togTheme {{ cb }}" means: "here is the recipe; run it on click."

A function that makes functions

The trickiest pattern in your file. Each checkbox needs its own click recipe, so one factory builds them:

const togCheck = (id) => () => {
  … toggle the checkbox whose name is `id` …
};

togCheck('p2a')   ← calling the factory RETURNS a recipe for checkbox 'p2a'
                    (nothing is toggled yet — the recipe runs on click)

Read (id) => () => … as: "give me an id, and I hand you back a zero-input function that toggles that id."

5 · Transforming lists with .map()

.map() takes an array and a function, runs the function on each item, and returns a new array of the results. It is how raw plan data becomes display-ready data. Real example (simplified) — the strategy chips on Home:

stRoutes: PLANS.home.strategy.focus.routes.map(r => ({
  label: r.label,                ← copy the text through
  bg: TONES[r.tone].bg,          ← look up the color for its tone
  fg: TONES[r.tone].fg,
  go: () => this.setState({ nav: 'plan', sel: r.go })   ← its click recipe
}))

In words: "for each route r in the data, produce an object with a label, its colors, and a click handler." The template then loops over stRoutes with sc-for and renders one chip per entry.

Syntax note: r => ({ … }) — the parentheses around the braces mean "this is an object being returned," not a function body. Forgetting them is a classic error.

6 · Making decisions

Comparison

S.nav === 'home'    ← true if nav equals 'home'  (=== compares, = assigns!)
S.nav !== 'home'    ← true if it does NOT
score >= 210        ← numeric comparisons
!S.sbOpen           ← ! flips a boolean: "not open"

The ternary — a one-line if/else

condition ? valueIfTrue : valueIfFalse. Your file uses it everywhere to pick styles:

theme === 'dark' ? 'Light mode' : 'Dark mode'   ← the toggle button's label
isMobile ? '1fr' : 'repeat(3, minmax(0, 1fr))'  ← 1 column on phone, 3 on desktop

Fallbacks with || and ??

localStorage.getItem('pf-track') || 'none'   ← use saved value, or 'none' if empty
this.props.dimStrength ?? 0.32               ← use the prop, or 0.32 if not set

Classic if

if (t) {
  this.setState({ pfTrack: t });   ← runs only when t is non-empty
}

Remember the template rule: none of this logic may live inside {{ ob }} … {{ cb }} holes. Decisions happen here in JavaScript; the template only receives the finished answer (like isHome: S.nav === 'home').

7 · State — the app's memory, and why the screen updates

State is one object holding everything that can change while you use the app:

state = {
  theme: 'dark',     ← which theme
  nav: 'home',       ← which page is open
  sbOpen: true,      ← sidebar expanded?
  checks: {          }  ← which boxes are ticked, e.g. { p2a: true }
};

The golden rule: never change it directly. Always go through this.setState(…), which does two things: updates the values, then automatically re-runs renderVals() so the screen redraws. Assigning directly (this.state.nav = 'plan') changes the memory but the screen never hears about it.

this.setState({ nav: 'plan' });   ← merge: only nav changes, rest stays

this.setState(s => ({ n: s.n + 1 }));   ← function form: when the new value
                                           depends on the current one

So every interaction in the dashboard is the same three beats: click → setState → re-render. Nothing else. No code ever "finds the element and changes its color" — the color simply depends on state, and follows it.

8 · Saving — JSON and localStorage

State lives in memory and dies on reload. To survive, it is written to localStorage — a tiny per-site key/value store in your browser. It only stores strings, so objects are converted with JSON:

JSON.stringify({ p2a: true })   → the string '{"p2a":true}'   (object → text)
JSON.parse('{"p2a":true}')      → the object back            (text → object)

Writing (inside the checkbox handler) and reading back (on load):

localStorage.setItem('strategy-dashboard-checks', JSON.stringify(next));

componentDidMount() {   ← runs once when the page opens
  const saved = JSON.parse(localStorage.getItem('strategy-dashboard-checks') || '{ }');
  this.setState({ checks: saved });
}

The try { … } catch (e) { … } wrappers around these in your file mean: "attempt this; if it fails (e.g. storage blocked), don't crash — just skip it."

9 · A real feature, end to end: ticking a checkbox

Everything above, working together. Follow one click through the whole system.

STEP 1 · THE TEMPLATE RENDERS THE BOX
<div onClick="{{ ob }} ut.togCheck {{ cb }}" style="… background:{{ ob }} ut.boxBg {{ cb }};">{{ ob }} ut.mark {{ cb }}</div>

Each today-item ut arrives from renderVals() already carrying its own click recipe (togCheck), its colors, and its ✓ mark.

STEP 2 · THE HANDLER RUNS ON CLICK
const togCheck = (id) => () => {
  const next = Object.assign({ }, this.state.checks, { [id]: !this.state.checks[id] });
  this.setState({ checks: next });
  try { localStorage.setItem('strategy-dashboard-checks', JSON.stringify(next)); }
  catch (err) { /* ignore */ }
};

Line by line: build next — a fresh copy of all checks (Object.assign = "copy these objects together into a new one") with this one id flipped (! = opposite of its current value). Then setState updates memory, and localStorage.setItem saves it as text.

STEP 3 · THE SCREEN FOLLOWS

setState triggers renderVals() again. It recomputes each item's boxBg/mark from the new checks, the holes refill, and the box appears ticked — here and on the plan page, because both read the same state.

STEP 4 · NEXT VISIT

componentDidMount() reads the saved string back, JSON.parse turns it into an object, setState restores it — your ticks are still there.

10 · Practice edits — safe first steps

Do these in order. Before each one, duplicate the file as a backup. If the page goes blank, you likely broke a comma, quote, or brace — undo and compare.

① Change a string. In plans(), find title: 'Study Abroad Plan' and reword it. Keep the quotes. If your text contains an apostrophe, use the curly kind (’) or it will end the string early.
② Add an item to an array. In an actions list, copy one { … }, task entry, paste it below, change its id (must be unique — it's the checkbox's save-name) and its text.
③ Change a decision. Find theme: 'dark' in state and make the default 'light'. One string, but now you've edited behavior.
④ Follow a value. Pick any text on screen, search the file for it, and trace: which plans() key holds it → which renderVals() line passes it on → which template hole displays it. Once you can do this, you can change anything.
